Class RTCPeer<EventMap>

A simple wrapper for RTCPeerConnection that implements perfect negotiation.

Usage example

 
const peer1 = new RTCPeer();
const peer2 = new RTCPeer();

peer1.addEventListener('candidate', (c) => peer2.handleCandidate(c));
peer2.addEventListener('candidate', (c) => peer1.handleCandidate(c));
peer1.addEventListener('description', (d) => peer2.handleDescription(d));
peer2.addEventListener('description', (d) => peer1.handleDescription(d));

peer1.media.addEventListener('track', ({ streams: [stream] }) => {
  const video = document.createElement('video');
  video.autoplay = true;
  video.srcObject = stream;
  document.body.appendChild(video);
});

const stream = await navigator.mediaDevices.getUserMedia({video: true});
peer2.media.addTrack(stream.getTracks()[0], stream);

Constructors

constructor

  • new RTCPeer(polite?: boolean | null): RTCPeer
    • Defined in src/rtc/peer/index.ts:67

    Constructs a new RTCPeer and managed RTCPeerConnection.

    Optionally, override whether the peer is "polite", i.e. if it would forget about its own offer and acknowlege an incoming offer when a connection negotiation collision occurs. By default, RTCPeer determines whether to be polite in perfect negotiation collisions by comparing its connection's session description (SDP) origin field (o=) against the other peer's offered SDP. To override this behavior, manually set whether the peer is polite via the parameter.

    Parameters

    • Default value polite: boolean | null = null

      Use to override whether this peer is polite when a negotiation collision occurs. If either no value or null are provided, the peer automatically compares SDP origins to determine politeness.

    Returns RTCPeer

  • handleCandidate(candidate: RTCIceCandidate): Promise<void>
    • Defined in src/rtc/peer/index.ts:91

    Handle a RTCIceCandidate from the associated RTCPeer, likely sent via a signaling server.

    Unless the RTCPeer is ignoring incoming candidates due to a negotiation collision (in which case, any errors are caught instead of bubbling up), this method does nothing special beyond adding the candidate to the underlying connection.

    Parameters

    • candidate: RTCIceCandidate

      the candidate sent from the associated peer

    Returns Promise<void>

handleDescription

  • handleDescription(description: RTCSessionDescription): Promise<void>
    • Defined in src/rtc/peer/index.ts:109

    Handle a RTCSessionDescription, or SDP, from the associated peer, likely sent via a signaling server.

    This method determines whether or not to acknowlege or ignore the descriptions that are passed to it, thus the majority of the negotiation logic can be found here.

    If an "answer" is necessary, i.e. if the incoming description is of type offer and we are not ignoring the peer's offer due to a collision, a RTCPeerDescriptionEvent will be emitted as the Promise resolves, to be sent to the other peer.

    Parameters

    • description: RTCSessionDescription

      the session description (SDP) sent from the associated peer

    Returns Promise<void>
